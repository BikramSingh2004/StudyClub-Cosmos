1. Project Overview & Objectives

1.1 What is Virtual Cosmos?

Virtual Cosmos is a 2D multiplayer virtual environment where users can move around freely and interact with each other in real time. The core mechanic is proximity-based chat:

  • When two users come close (within a 150-pixel radius) → a chat channel automatically connects between them
  • When users move apart (beyond the radius) → the chat channel automatically disconnects

This simulates real-world spatial interaction — just like in a physical room, you can only talk to people who are near you. The experience is similar to platforms like Gather.town or Cosmos.video.

2.2 Data Flow

There are three primary data flows in the system:

Flow 1: Position Synchronization

Client captures keyboard input → calculates new position → emits player:move to server → server updates in-memory state → broadcasts player:moved to all other clients → other clients update the PixiJS sprite position.

Flow 2: Proximity Detection

On every player:move event, the server iterates through all active players, calculates Euclidean distance. If distance < 150px and not already connected → joins both sockets to a shared room and emits chat:connected. If distance ≥ 150px and currently connected → removes from room and emits chat:disconnected.

Flow 3: Chat Messaging

When connected, client sends chat:message with room ID and content → server persists to MongoDB → broadcasts the message to all sockets in that room → both clients display the message in their chat panels.

5.1 Server Setup (server/index.js)

The server creates an HTTP server with Express, then attaches Socket.IO to it:

const app = express();
const server = http.createServer(app);
const io = new Server(server, {
  cors: {

    methods: ['GET', 'POST'],
  },

CORS is configured to accept connections from the Vite dev server. In production, this would be set to the deployed frontend URL.

5.2 In-Memory State Management

The server maintains two critical data structures in memory for real-time performance:

// Map: socketId → { userId, username, x, y, avatarColor }
const activePlayers = new Map();

// Set of active chat connections: "socketId1::socketId2" (sorted)
const activeConnections = new Set();
Why In-Memory?

Database queries on every position update (potentially 30 times/second/player) would create unacceptable latency. The in-memory Map provides O(1) lookups for player positions. MongoDB is used only for persistence (user profiles and chat history), not real-time state.

activePlayers Map Structure

KeyTypeDescription
socketIdString (Map key)Unique Socket.IO connection identifier
userIdStringSame as socketId (derived from connection)
usernameStringPlayer's chosen display name
xNumberCurrent X position in the 2D world
yNumberCurrent Y position in the 2D world
avatarColorString (hex)Avatar color (e.g., "#6366f1")

activeConnections Set Structure

Each entry is a string in the format "socketId1::socketId2", where the IDs are alphabetically sorted. This ensures a consistent key regardless of which player moved — getRoomId("abc", "xyz") and getRoomId("xyz", "abc") both return "abc::xyz".

function getRoomId(id1, id2) {
  return [id1, id2].sort().join('::');
}

5.3 Player Lifecycle Events

player:join — User Enters the Cosmos

socket.on('player:join', async (data) => {
  const { username, avatarColor } = data;

  // Random spawn position within world bounds
  const x = 100 + Math.random() * 1800;
  const y = 100 + Math.random() * 800;

  // Store in memory
  activePlayers.set(socket.id, { userId: socket.id, username, x, y, avatarColor });

  // Persist to MongoDB (upsert = create or update)
  await User.findOneAndUpdate(
    { username },
    { avatarColor, lastPosition: { x, y }, lastActive: new Date() },
    { upsert: true, new: true }
  );

  // Send current state to joining player
  socket.emit('player:current', Object.fromEntries(activePlayers));

  // Notify all other players
  socket.broadcast.emit('player:joined', { socketId: socket.id, ...playerData });

  // Check if spawned near anyone
  checkProximity(socket.id);
});

player:move — Position Update

socket.on('player:move', (data) => {
  const player = activePlayers.get(socket.id);
  if (!player) return;

  player.x = data.x;
  player.y = data.y;

  // Broadcast to everyone else
  socket.broadcast.emit('player:moved', { socketId: socket.id, x: data.x, y: data.y });

  // Re-evaluate all proximity relationships
  checkProximity(socket.id);
});

disconnect — User Leaves

When a player disconnects, the server:

  1. Finds and removes all active chat connections involving this player
  2. Notifies the other side of each connection with chat:disconnected
  3. Saves the player's last position to MongoDB
  4. Removes the player from the in-memory Map
  5. Broadcasts player:left to all remaining players

6.3.4 Keyboard Input & Movement

Input uses a Set to track currently pressed keys, allowing simultaneous key presses (diagonal movement):

// In the game loop (ticker callback):
const keys = keysRef.current;
let dx = 0, dy = 0;

if (keys.has('w') || keys.has('arrowup'))    dy -= MOVE_SPEED;
if (keys.has('s') || keys.has('arrowdown'))  dy += MOVE_SPEED;
if (keys.has('a') || keys.has('arrowleft'))  dx -= MOVE_SPEED;
if (keys.has('d') || keys.has('arrowright')) dx += MOVE_SPEED;

// Diagonal normalization: prevents faster diagonal movement
if (dx !== 0 && dy !== 0) {
  const factor = Math.SQRT1_2;  // ≈ 0.707
  dx *= factor;
  dy *= factor;
}
Diagonal Normalization

Without normalization, moving diagonally (pressing W+D) would result in a speed of √(4² + 4²) ≈ 5.66px/frame — 41% faster than cardinal movement. Multiplying by √(1/2) ≈ 0.707 brings diagonal speed back to exactly 4px/frame, ensuring uniform movement speed in all directions.

6.3.5 Camera System

// Smooth camera follow with lerp (linear interpolation)
const targetX = -localPosRef.current.x + window.innerWidth / 2;
const targetY = -localPosRef.current.y + window.innerHeight / 2;
world.x += (targetX - world.x) * 0.1;  // 10% lerp factor
world.y += (targetY - world.y) * 0.1;

The camera doesn't jump to the player position — it smoothly interpolates at 10% per frame, creating a fluid following effect. The world container itself is translated (the viewport stays fixed), so the local player always appears near the center of the screen.

6.3.6 Position Emit Throttling

// Throttle to ~30fps (33ms intervals)
const now = Date.now();
if (now - lastEmit > 33) {
  socket.emit('player:move', { x: newX, y: newY });
  lastEmit = now;
}

While the game loop runs at 60fps, network position updates are throttled to ~30fps. This reduces bandwidth by 50% while remaining visually smooth, since remote avatars are updated at each received position rather than interpolated.

7. Proximity Detection — Core Logic

7.1 The Algorithm

Proximity detection is the core mechanic of Virtual Cosmos. It runs on the server to prevent cheating and ensure consistency. The algorithm executes every time any player moves.

const PROXIMITY_RADIUS = 150; // pixels

function getDistance(a, b) {
  return Math.sqrt((a.x - b.x) ** 2 + (a.y - b.y) ** 2);
}

function checkProximity(movedSocketId) {
  const movedPlayer = activePlayers.get(movedSocketId);
  if (!movedPlayer) return;

  for (const [otherSocketId, otherPlayer] of activePlayers) {
    if (otherSocketId === movedSocketId) continue;

    const dist = getDistance(movedPlayer, otherPlayer);
    const roomId = getRoomId(movedSocketId, otherSocketId);
    const isConnected = activeConnections.has(roomId);

    if (dist < PROXIMITY_RADIUS && !isConnected) {
      // ── CONNECT ──
      activeConnections.add(roomId);
      // Join both sockets to shared Socket.IO room
      io.sockets.sockets.get(movedSocketId)?.join(roomId);
      io.sockets.sockets.get(otherSocketId)?.join(roomId);
      // Load chat history from MongoDB and notify both clients
      // ... emit 'chat:connected' to both
    }
    else if (dist >= PROXIMITY_RADIUS && isConnected) {
      // ── DISCONNECT ──
      activeConnections.delete(roomId);
      // Remove both from the room
      io.sockets.sockets.get(movedSocketId)?.leave(roomId);
      io.sockets.sockets.get(otherSocketId)?.leave(roomId);
      // Notify both clients
      // ... emit 'chat:disconnected' to both
    }
  }
}

7.2 Distance Formula

The Euclidean distance formula is used:

d = √( (x₂ - x₁)² + (y₂ - y₁)² ) Example: Player A at (200, 300) Player B at (300, 350) d = √( (300-200)² + (350-300)² ) d = √( 10000 + 2500 ) d = √12500 d ≈ 111.8 pixels 111.8 < 150 → CONNECTED ✓

7.3 Connection State Machine

┌─────────────┐ │ DISCONNECTED │ │ (default) │ └──────┬──────┘ │ distance < 150px │ ┌──────▼──────┐ │ CONNECTED │ ←── Chat room created │ (chatting) │ Messages flow └──────┬──────┘ │ distance ≥ 150px OR player disconnects │ ┌──────▼──────┐ │ DISCONNECTED │ ←── Chat room destroyed │ (default) │ Chat panel hides └─────────────┘

7.4 Complexity Analysis

MetricValueNotes
Per-move check O(n) Iterates through all other players. For n=100 players, that's 99 distance calculations per move.
Distance calc O(1) Two subtractions, two multiplications, one addition, one sqrt.
Connection lookup O(1) Set.has() on the roomId string — hash-based constant time.
Total per move O(n) Scales linearly with player count. Efficient for hundreds of players.

7.5 Visual Representation

On the client, the local player's proximity radius is visualized as:

  • A semi-transparent filled circle (3% opacity) showing the detection zone
  • A dashed ring border (25% opacity) at the exact radius boundary

This provides clear visual feedback so users understand when they're about to connect or disconnect.

8. Chat System — Design & Flow

8.1 Room-Based Architecture

Chat rooms are managed using Socket.IO's built-in room system. Each pair of connected players gets a unique room ID:

// Room ID = sorted socket IDs joined with "::"
// e.g., "abc123::xyz789"
const roomId = getRoomId(socketId1, socketId2);

// Join both players to the room
io.sockets.sockets.get(socketId1)?.join(roomId);
io.sockets.sockets.get(socketId2)?.join(roomId);

// Messages sent to the room reach both players
io.to(roomId).emit('chat:message', message);

8.2 Connection Lifecycle

1

Proximity Detected (server-side)

Distance check triggers. Both socket IDs are joined to a shared room. The server loads the last 50 messages from MongoDB for this room ID.

2

chat:connected emitted to both clients

Each client receives: peer's name, color, socket ID, room ID, and message history array. The ChatPanel component adds this to its chats Map state.

3

Chat Panel Appears

React re-renders. The ChatPanel detects chats.size > 0 and renders. If multiple chats exist, tabs appear for each connected peer.

4

Messaging

User types a message → client emits chat:message { roomId, content } → server persists to MongoDB → server broadcasts to the room → both clients display the message.

5

Proximity Lost (server-side)

Distance check triggers disconnect. Both sockets leave the room. chat:disconnected emitted. Client removes the peer from the chats Map. If no chats remain, the panel hides.

When a chat connects, the server queries the last 50 messages for that room and sends them as history. This means if two users reconnect (walk away then return), their previous conversation is preserved.

10. Database Schema & Design

10.1 Users Collection

// server/models/User.js
{
  _id: ObjectId,
  username: { type: String, required: true, unique: true, trim: true },
  avatarColor: { type: String, default: '#6366f1' },
  lastPosition: {
    x: { type: Number, default: 400 },
    y: { type: Number, default: 300 }
  },
  lastActive: { type: Date, default: Date.now },
  createdAt: Date,   // auto (timestamps: true)

}

Purpose: Persists user profiles across sessions. Tracks the last known position and activity time. Uses upsert on join, so returning users are updated rather than duplicated.

10.2 Messages Collection

// server/models/Message.js
{
  _id: ObjectId,
  sender: { type: String, required: true },
  senderColor: { type: String, default: '#6366f1' },
  content: { type: String, required: true },
  roomId: { type: String, required: true, index: true },
  timestamp: { type: Date, default: Date.now }
}

Purpose: Persists all chat messages. The roomId field is indexed for fast history lookups when users reconnect. Messages are queried with .sort({ timestamp: -1 }).limit(50) and reversed for chronological order.

10.3 Graceful Degradation

MongoDB is Optional

The server catches MongoDB connection errors and logs a warning. If MongoDB is unavailable, the app still works fully — movement, proximity detection, and real-time chat all function. Only message persistence and user profile storage are affected.

  • Automatic behavior: Chat connects/disconnects automatically — no manual action needed.
  • Visual feedback: Green pulse dots for "online" status. Animated elements for active connections.
  • Responsive: Canvas and UI resize with the window. Panels use fixed positioning.

The PROXIMITY_RADIUS value must be the same on both client and server. The client uses it for the visual ring; the server uses it for the actual detection. If they differ, the visual feedback will not match the actual connection behavior.

15. Security Considerations

ConcernHow Addressed
Server-side proximity Proximity detection runs on the server, not the client. Users cannot fake being near someone to force a chat connection.
CORS Socket.IO and Express are configured to only accept connections from the specified client URL.
Input validation Chat messages are trimmed and checked for empty content before being saved or broadcast. Username is trimmed and limited to 20 characters on the client.
Room isolation Chat messages are only sent to the specific Socket.IO room. A player cannot receive messages from rooms they haven't been added to by the server.
Position bounds Client-side position clamping prevents users from moving outside the world. Server could add server-side validation for additional security.
Environment variables Database credentials and configuration stored in .env, excluded from git via .gitignore.
